# GitHub 형상관리 전략

## 원칙
- 문서 구조 변경은 기능 변경과 분리해 추적 가능하게 유지합니다.
- 루트 인덱스와 current 허브 변경 시 함께 검토합니다.
- 기능명세 이동/개명 시 링크와 진입점을 동시에 갱신합니다.

## 대상 저장소
- 백엔드: `https://github.com/MINJI-AI-STUDY/AI-BACK`
- 프론트엔드: `https://github.com/MINJI-AI-STUDY/AI-FRONT`
- 기본 브랜치는 항상 `main`입니다.

## 브랜치 전략
- GitHub Flow를 사용합니다.
- 모든 작업은 `main`에서 분기한 작업 브랜치에서 수행합니다.
- 직접 `main`에 push 하지 않습니다.
- 모든 변경은 PR로만 `main`에 merge 합니다.
- merge 후 로컬 작업 브랜치는 삭제합니다.
- 원칙적으로 `1기능 = 1브랜치 = 1PR`로 운영합니다.

## 작업 단계 규칙
1. 명령 수신
2. 명령 세분화
3. 계획 수립
4. 계획 구현
5. Oracle 리뷰
6. 커밋/푸시를 작은 단위로 분리
7. PR 생성
8. 인라인 리뷰 및 코멘트 확인
9. 리뷰 코멘트 반영 후 재수정 또는 merge
10. merge 후 로컬 브랜치 삭제

## 커밋/PR 언어 규칙
- 모든 커밋 메시지는 한글로 작성합니다.
- 모든 PR 제목/본문/코멘트는 한글로 작성합니다.
- 리뷰 코멘트 답변도 한글로 작성합니다.

## 커밋 전략
- 프론트/백엔드/AI/문서 변경은 가능한 한 분리 커밋합니다.
- 하나의 커밋은 하나의 의도만 담습니다.
- 환경변수 파일, 비밀키, 인증정보는 커밋하지 않습니다.
- `.env`, `.env.local`, 인증서, 개인 토큰 파일은 절대 올리지 않습니다.

## PR 크기 전략
- PR은 가급적 1000줄 이하로 유지합니다.
- 1000줄을 넘길 가능성이 있으면 기능을 더 작게 쪼개 브랜치와 PR을 분리합니다.
- 리뷰어가 한 번에 이해할 수 있는 범위로 잘라 올립니다.

## PR 전략
- PR은 작은 범위로 나눠 올립니다.
- PR 본문에는 변경 이유, 영향 범위, 검증 결과를 반드시 적습니다.
- Oracle 리뷰가 필요한 변경은 PR 전 또는 PR 직후 반드시 검토를 거칩니다.
- 인라인 리뷰가 오면 코멘트 단위로 반영 여부를 명확히 남깁니다.

## 리뷰 코멘트 원칙
- PR 인라인 코멘트와 리뷰는 사람처럼 실제 의견을 남깁니다.
- 형식적 칭찬보다 대안 제시와 우려 제기를 우선합니다.
- 예: "A도 가능하지만 이 경우엔 B가 더 낫지 않나요? 이유는 ..." 같은 식의 실제 리뷰를 남깁니다.
- 근거 없는 칭찬성 코멘트는 지양합니다.

## 리뷰 반영 플로우
1. 인라인 리뷰 코멘트 확인
2. 코멘트별로 수용/반박/보류 중 하나를 결정
3. 수용 시 수정 커밋으로 반영
4. 반박 시 기술적 이유를 한글로 명확히 답변
5. 더 나은 대안이 있으면 대안을 반영하고 이유를 남김
6. 최종적으로 모든 코멘트가 처리된 뒤 merge 진행
