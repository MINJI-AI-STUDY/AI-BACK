# ADR-004 학교-학급 논리 멀티테넌시

## 결정
- 백엔드는 물리적 분리 대신 **논리적 멀티테넌시**를 적용합니다.
- 최상위 테넌트는 `schoolId`로 정의합니다.
- 학교 하위 조직은 `classroomId`로 정의합니다.
- 모든 사용자/자료/문제세트/제출/대시보드/질의응답 로그는 최소 `schoolId`를 가져야 합니다.
- 학급 맥락이 필요한 데이터는 `classroomId`를 추가로 가집니다.

## 이유
- 현재 MVP 규모에서는 저장소를 학교 단위로 논리 분리하는 것이 운영 효율과 구현 비용의 균형이 가장 좋습니다.
- 학급 단위 통계/배포/권한 제어를 지원하려면 학교 하위 조직 식별자가 필요합니다.

## 적용 원칙
- API 인증 후 사용자 컨텍스트에서 `schoolId`, `classroomId`를 복원합니다.
- 조회/집계는 항상 같은 `schoolId` 내부로 제한합니다.
- 교사 기능은 자신의 `schoolId` 범위에서만 자료/문제/대시보드를 다룹니다.
- 학생 기능은 자신의 `schoolId`, `classroomId` 맥락에서만 문제풀이/결과/QA를 수행합니다.
- 운영자 overview는 학교 간 전체 요약도 가능하지만, 기본 단위는 학교 기준 집계입니다.

## 프론트 책임
- 프론트는 tenant를 임의로 선택하지 않고, 로그인된 사용자 컨텍스트를 기준으로 동작합니다.
- 학교/학급 정보는 화면 상단 컨텍스트나 메타 정보로 표시할 수 있으나, 권한 판단은 백엔드 응답을 기준으로 합니다.

## 후속 구현 포인트
- 사용자 모델에 `schoolId`, `classroomId` 추가
- 자료/문제세트/제출/QALog 모델에 `schoolId` 추가
- 대시보드 집계에 school filter 강제
- 필요 시 학교별 브랜딩/테마 확장 가능
